## Fixes — Priority 1 (correctness gap)

### 1. Missing `ALIGN:SAMTOOLS_FLAGSTAT` resource block
`conf/process.config:49` only sets resources for `PREPROCESS:SAMTOOLS_FLAGSTAT`. `subworkflows/align.nf:68` calls the same process under `ALIGN:` scope — it silently falls back to Nextflow defaults (1 cpu, no mem/time cap), risking OOM/timeout kills on large merged BAMs.

**Fix:** add after line 49:
```groovy
withName: 'ALIGN:SAMTOOLS_FLAGSTAT' { cpus = 4; memory = 8.GB; time = 4.h }
```

## Fixes — Priority 2 (naming)

### 2. CRAMINO scope-naming breaks its own convention
- PREPROCESS scope: `CRAMINO_RAW`, `CRAMINO_SEGMENTED`, `CRAMINO_FILTERED`, `CRAMINO_FL`, `CRAMINO_FLTNC` — suffix = pipeline stage.
- ALIGN scope: plain `CRAMINO` — no suffix, breaks pattern, ambiguous in logs (see user's run output: `ALIGN:CRAMINO` reads odd next to five suffixed siblings).

**Fix:** rename alias to `CRAMINO_ALIGNED` in `subworkflows/align.nf:8` import and `conf/process.config:50` withName key. One rename, two files.

### 3. `SAMTOOLS_FLAGSTAT` reused across scopes with only one config entry
Same root cause as #1. Once #1 is fixed this resolves itself — but flag it as a **pattern**: any process imported into 2+ subworkflow scopes needs a `withName` entry per scope, not one. Worth a CLAUDE.md dev-guideline line.

### 4. Process name prefixing is inconsistent
Some processes carry scope prefixes (`PREPROCESS:SKERA_SPLIT`), others don't appear in any scope (`COLLECT_INSTRUMENT_STATS`, `MULTIQC` — correct, these are pipeline-level singletons). No actual bug, just worth a one-line dev-guideline note: "scope prefix = process runs inside that subworkflow; no prefix = pipeline-level, single invocation."

## Fixes — Priority 3 (overcomplicated / duplicated)

### 5. Six `withName` blocks for one process (CRAMINO) instead of a parameterized resource tier
`conf/process.config:35-39,50`. Combined with #2, collapsing the naming also gives a chance to dedupe resource blocks — all five PREPROCESS CRAMINO variants use near-identical cpu/mem/time. Pull repeated `8 cpus / 16GB / 6h`, `4 cpus / 8GB / 4h` etc. into named tiers at top of `conf/process.config`:

```groovy
def small   = [cpus: 1,  memory: 2.GB,  time: 30.m]
def std     = [cpus: 4,  memory: 8.GB,  time: 4.h]
def large   = [cpus: 8,  memory: 16.GB, time: 6.h]
def xlarge  = [cpus: 16, memory: 32.GB, time: 12.h]
```
Then `withName: '...' { cpus = std.cpus; memory = std.memory; time = std.time }`. Cuts ~10 duplicate blocks down to lookups. yagni-check: only do this if you're touching process.config again soon — pure config rewording isn't worth a standalone PR.

### 6. `CB_SUFFIX_INJECT` AWK one-liner is unreadable in-place
`modules/align.nf:198-201` — multi-branch AWK embedded in process script block, no test coverage. Pull into `bin/inject_cb_suffix.awk`, call as a file. Easier to read, diff, and unit-test in isolation.

### 7. Repeated standard/custom barcode branch-then-remix pattern
`subworkflows/align.nf:27-53` branches into `.standard`/`.custom`, maps each separately, then `.mix()`es back. Works, but the two `.map` blocks are 90% identical (only the whitelist source differs). Collapse to one `.map` after picking the whitelist channel per-item, e.g. precompute `ch_whitelist_resolved` then single `.map`. shrink, not rewrite — same logic, less surface.

## Subworkflow & process rename proposal

Read all 5 subworkflow files end to end. Current names are functional but stage names leak implementation detail (FL, FLTNC, CB, GTF) instead of pipeline meaning. Proposal below — rename only, no logic change, do in one PR per subworkflow so diffs stay reviewable.

### General rules to apply everywhere
1. **Stage suffix = what happened, not the tool's internal acronym.** `CRAMINO_FLTNC` tells you which cramino invocation, not what state the reads are in. Rename to lifecycle stage: `raw → segmented → length_filtered → full_length → trimmed` (FLTNC = "full-length, trimmed, non-concatemer" — write that out once in CLAUDE.md, then suffix as `_trimmed`).
2. **One resource-tier suffix convention per scope**, applied to every multi-instance process, not just CRAMINO (already flagged as #2 — this generalizes it).
3. **Subworkflow names describe pipeline stages, not tool names.** `CLASSIFY` is fine; `ALIGN` undersells what it does (whitelist correction + dedup + alignment + CRAM archive — alignment is step 4 of 5).

### Subworkflow-by-subworkflow

**`subworkflows/preprocess.nf` → keep name, rename internals**
- CRAMINO aliases: `CRAMINO_RAW, CRAMINO_SEGMENTED, CRAMINO_FILTERED, CRAMINO_FL, CRAMINO_FLTNC` → `CRAMINO_RAW, CRAMINO_SEGMENTED, CRAMINO_LENGTH_FILTERED, CRAMINO_FULL_LENGTH, CRAMINO_TRIMMED`. Matches actual variable/output names already used in-file (`filtered_bam`, `fl_bam`, `fltnc_bam` at lines 47, 113, 127) — currently the CRAMINO alias suffixes don't even match the channel-variable vocabulary in the same file. This is the actual root inconsistency, bigger than the ALIGN-scope gap noted in fix #2.
- `ch_branched_segmented` (line 73) → `ch_index_branch` — "segmented" already happened upstream of branching; current name implies the branch *produces* segmentation.
- `isoquant_shards()` lives in classify.nf but conceptually is genome-partition config, not classify logic — see CLASSIFY note below.

**`subworkflows/align.nf` → rename to `barcode_align.nf` / workflow `BARCODE_ALIGN`**
- Current name collides conceptually with PBMM2_ALIGN (one process inside it) while the subworkflow itself does 5 things: merge → whitelist correct → dedup → align → CRAM. "ALIGN" undersells the barcode-correction half, which is arguably the harder logic (lines 26-53). Rename workflow to `BARCODE_ALIGN`, file to `subworkflows/barcode_align.nf`.
- `CRAMINO` (bare, line 8/77) → `CRAMINO_ALIGNED` (already fix #2, restated here for completeness).
- `ch_split_merged` (line 32) → `ch_barcode_source_branch` (standard whitelist vs custom reverse-complemented whitelist branch — "split_merged" reads like a BAM-splitting step, it's actually a whitelist-source branch).
- `ch_standard_correct_input` / `ch_custom_correct_input` (lines 42, 48) → fine as-is, already descriptive.

**`subworkflows/classify.nf` → rename workflow to `ISOFORM_CLASSIFY`, keep file name**
- `isoquant_shards()` (lines 11-31) → move to a `conf/genome_shards.config` or `lib/` helper, rename to `genomeShards()` or `chromosomeShardMap()`. Reasoning: it's static genome-partition data, unrelated to classify *logic*; currently buried at the top of a 128-line workflow file makes it easy to miss when porting to a non-human genome.
- `CB_SUFFIX_INJECT` → fine, but the variable it produces `ch_with_suffix` (line 50) → `ch_cb_disambiguated` — "suffix" alone doesn't say *why* (avoiding cell-barcode collisions across libraries in joint experiment-level calling).
- `SPLIT_BAM_BY_SHARD` → fine.
- Step comments (lines 38, 52, 55, 63, 74, 80, 95, 101) are already good — best-documented file in the repo, use as the house style for the others.

**`subworkflows/quantify.nf` → keep as-is.** 18 lines, single process, clear. No changes.

**`subworkflows/export.nf` → keep as-is.** 33 lines, clear 3-step structure. No changes.

### Net effect
- 1 file rename (`align.nf` → `barcode_align.nf`, workflow `ALIGN` → `BARCODE_ALIGN`) — touches `main.nf` call site too.
- ~6 CRAMINO alias renames across 2 subworkflow files + `conf/process.config` withName keys.
- 1 helper function relocated out of `classify.nf` into config/lib.
- 3 local channel-variable renames for clarity, zero behavior change.

Do this as a single rename-only PR, no logic touched — keeps `git diff` reviewable and rollback trivial if CI naming-dependent anything breaks (check `MULTIQC` config doesn't pattern-match on old process names first).
